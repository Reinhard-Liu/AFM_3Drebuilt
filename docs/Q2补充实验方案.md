### 2.4 当前结果的核心判断

这组结果说明：

1. 当前模型最强的是：
   - 中心定位
   - 杂原子识别
   - z 轴恢复
   - 检索支持
   - 容忍轻微错位后的局部邻接恢复
2. 当前模型最弱的是：
   - strict typed graph 闭环恢复
   - 预测对象条件下的类型和边迁移

3. 因此当前阶段最合理的实验策略不是立刻继续大改架构，而是：
   - 先把现有主张的 full-test 证据补齐
   - 再通过 targeted ablation 和 error decomposition 判断是否值得继续大改

## 3. 新实验设计原则

### 3.1 先冻结主模型，再补证据

当前推荐先把 `v20 epoch 10` 暂时视为**论文主模型**，先完成 formal evaluation。

原因：

- 你已经有一套较完整的最新模型与结果
- 当前最缺的是 full-test 证据，而不是又一轮未验证的新结构
- 如果 full-test 一做就发现主张站不住，再继续改架构也更有方向

### 3.2 所有新实验必须区分三层证据

所有表格与结论必须明确属于哪一层：

1. validation
2. fixed-15 qualitative set
3. full-test formal benchmark

严禁混写。

### 3.3 所有主表必须同时报告 strict 和 robust 指标

这是当前版本最关键的实验设计原则。

如果只报 strict 指标，会低估当前模型已经学到的局部结构能力；如果只报 robust 指标，又会掩盖对象错位问题。

因此正式结果必须同时给出：

- strict typed-graph 指标
- robust local-geometry / local-edge 指标

### 3.4 什么时候才应该继续大改模型

只有在完成下面两类实验后，才建议继续大改模型：

1. full-test formal benchmark
2. type/edge gap decomposition

如果这两类实验表明：

- robust 指标也不高
- 类型错误不是由对象错位主导

那才说明需要继续重做对象级 type/edge 分支。

### EXP-01：v20 epoch10 全测试集正式对象级 benchmark

#### 优先级

`P0`

#### 实验目的

把当前 `validation` 结果升级成 full-test 正式结果，确定当前模型是否真的可以作为论文主模型。

#### 需要回答的问题

1. `peak_object_score = 0.8170` 是否能在 full-test 上稳定成立？
2. `pred_object_score = 0.7129` 是否只是 validation 偶然值？
3. `peak-center` 与 `pred-object` 的差距在 full-test 上到底有多大？

#### 必须输出的指标

- `peak_object_score`
- `gt_object_score`
- `pred_object_score`
- `peak_center_type_acc`
- `peak_center_macro_f1`
- `peak_center_hetero_f1`
- `peak_center_edge_f1`
- `pred_object_type_acc`

- `peak_center_shift_px`

#### 建议新增输出

- full-test 均值 / 标准差
- best / median / worst case
- gap:
  - `gt_object_score - peak_object_score`
  - `peak_object_score - pred_object_score`

#### 输出结果应达到什么效果

- 如果 full-test 明显接近 validation，当前 `v20 epoch10` 可以继续当论文主模型

#### 优先级

`P0`

- `pred_object_type_acc`
- 新增：
  - `pred_object_type_acc_matched_r3`
  - `pred_object_macro_f1_matched_r3`
  - `pred_object_pair_dist_mae_matched`
  - `pred_object_bond_len_mae_matched`

#### 必须回答的问题

1. 当前 strict 边损失中，有多少来自中心错位？

### SUP-02：graph reconstruction 风格 baseline

#### 优先级

`P0`

#### 实验目的

补一个非 dense 的图重建路线对照，避免审稿人认为当前方法只是优于“像素图翻译”，而不是优于“图恢复”本身。

#### baseline 定位

这组 baseline 不要求完全复现外部论文，但必须满足：

- 输入仍是 AFM 堆栈
- 中间表征以 graph / node-edge reconstruction 为主
- 输出可以转成与 `v20` 相同的对象级 typed graph 与 `2D+z` 评估口径

#### 推荐实现方向

1. `AFM -> node proposals -> graph decoder`
2. `AFM encoder -> set decoder -> node type + edge adjacency`
3. 如实现成本过高，可先做轻量 graph-head baseline：
   - 不用当前 center-conditioned type head
   - 直接从 proposal set 上做 node-edge 联合图预测

#### 必须回答的问题

1. graph 风格 baseline 的类型能力是否优于 dense baseline
2. 它是否能逼近当前对象级路线，还是仍明显落后
3. 当前对象级路线的优势到底来自：
   - center curriculum
   - `2D+z` 中间表征

## 8. 什么时候应该继续改模型

只有在下面两种情况之一成立时，才建议你重新进入大规模架构迭代：

1. `EXP-01` 表明 full-test 结果明显低于当前 validation 印象
2. `EXP-03/07` 表明 robust 指标也不足，说明模型并非只是对象错位，而是类型/边分支本身确实没学会

如果不是这两种情况，则当前更推荐：

- 先完成论文证据包
- 再有针对性地补 type/edge 迁移损失
- 同时优先完成 `SUP-01~03`，而不是分散到 `EXP-08~11`

## 9. 当前最重要的结论

基于现有结果，我的判断是：

1. 当前 v20 已经足以支撑一篇以**对象级 `2D+z` 中间重建**为核心的 `JMGM` 方法论文。
2. 当前还不足以支撑一篇以**高精度完整 typed 3D graph 恢复**为核心卖点的论文。
3. 论文最稳的主线应是：
   - 多高度 AFM
   - 对象级 `2D+z`
   - predicted-center curriculum
   - strict / robust 双评估
   - retrieval support
   - interpretable failure analysis

这条主线和当前真实结果是一致的，也最有机会写成一篇稳的投稿稿件。
