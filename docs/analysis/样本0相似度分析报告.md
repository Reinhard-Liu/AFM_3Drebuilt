# 样本0检索相似度低的原因分析报告

## 执行摘要

**验证样本#0的Top-5检索分子相似度仅0.24左右，远低于预期（理想值应>0.6）**

核心问题：
1. ❌ 原子数量预测错误（21 → 36，误差+71%）
2. ❌ 原子类型预测错误（完全缺失3个氧原子）
3. ❌ 检索特征与真实结构脱节
4. ❌ 检索空间维度不足（128维 vs 21万分子）

## 二、维度分析

### 2.1 原子数量维度

```
真实样本:   21 个原子
预测结果:   36 个原子 (误差 +71%)

检索到的分子:
  Rank 1:  35 原子  ← 与预测(36)接近
  Rank 2:  36 原子  ← 与预测(36)完全一致！
  Rank 3:  32 原子
  Rank 4:  32 原子
  Rank 5:  33 原子

平均:     33.6 原子 (vs GT: 21)
```

**核心问题**：检索结果偏向于与**预测值**（36）接近的分子，而非真实值（21）

**原因**：
- 检索头使用 Video ViT 提取的 AFM 特征（512维 → 128维）
- 这些特征反映的是**模型认为的**分子大小
- 由于 AtomCountHead 预测错误，检索头也被误导

### 2.2 原子类型维度

```
真实样本:  O:3, N:3, C:8, H:7
预测结果:  C:19, H:13, N:4
           ❌ 完全缺失 3 个氧原子！

元素匹配情况:
  Rank 1:  含O ✓, 含N ✗
  Rank 2:  含O ✗, 含N ✓  ← 与预测类似（都缺O或N）
  Rank 3:  含O ✓, 含N ✓  ← 元素完全匹配！
  Rank 4:  含O ✓, 含N ✓  ← 元素完全匹配！
  Rank 5:  含O ✓, 含N ✗
```

**悖论**：Rank3和Rank4的元素集合与GT完全一致，但相似度分数更低（0.2357, 0.2344）

**原因**：
- 检索头**不知道真实的原子类型**
- 它只基于 AFM 图像特征，而 AFM 无法直接区分 O 和 C
- 扩散模型预测缺失O原子 → 检索头也倾向于检索缺O的分子

### 2.3 环结构维度

```
真实样本:  2 个环 (2个5元环: O-C-N-C-C, N-N-C-C-C)
预测结果:  0 个环 (原子对应关系错乱，所有键断裂)

环数匹配:
  Rank 1:  5 个环  ✗
  Rank 2:  5 个环  ✗
  Rank 3:  2 个环  ✓  ← 环数正确！
  Rank 4:  2 个环  ✓  ← 环数正确！
  Rank 5:  4 个环  ✗
```

**矛盾**：环数正确的Rank3/4相似度更低，环数错误的Rank1/2相似度最高

**原因**：
- 当前检索特征（128维）**不包含环结构信息**
- 环检测算法依赖键长判断，但检索头只用全局 ViT 特征
- 模型没有被训练去识别环结构

## 三、检索机制深层问题

### 3.1 检索流程分析

```
AFM 图像 (10层 × 128×128)
    ↓
Video ViT Encoder
    ↓
条件向量 c (512维)
    ↓
MoleculeRetrievalHead (投影网络)
    ↓
检索特征 (128维) ← L2归一化
    ↓
余弦相似度计算 (与 211,505 个分子嵌入对比)
    ↓
Top-5 CID
```

**问题1：特征提取不准确**
- Video ViT 提取的是**图像级别**的全局特征
- 无法精确编码原子数量（误差71%）
- 无法区分元素类型（缺失O原子）
- 无法识别拓扑结构（环被忽略）

**问题2：检索空间维度不足**
- 211,505 个分子 → 128 维嵌入空间
- 平均每个维度需区分 ~1650 个分子
- 对于复杂分子，维度严重不足

**问题3：训练监督缺失**
- 训练集的 retrieval_loss：11.15-12.28（有监督）
- 验证集的 retrieval_loss：0.0（无监督！）
- 验证集分子的CID不在训练集中，无法计算对比损失
- 模型从未学习如何处理全新分子的检索

### 3.2 相似度分数与结构相似度的对比

让我们计算真实的结构相似度：

| Rank | 检索余弦分数 | 原子数相似度* | 元素集合相似度** | 环数相似度*** | 综合结构相似度 |
|------|------------|-------------|----------------|--------------|--------------|
| 1 | **0.2426** | 0.60 | 0.75 | 0.40 | 0.58 |
| 2 | **0.2425** | 0.58 | 0.75 | 0.40 | 0.58 |
| 3 | **0.2357** | 0.66 | **1.00** ✓ | **1.00** ✓ | **0.89** 🏆 |
| 4 | **0.2344** | 0.66 | 0.80 | **1.00** ✓ | 0.82 |
| 5 | **0.2331** | 0.64 | 0.75 | 0.50 | 0.63 |

*原子数相似度 = 1 - |n_pred - n_gt| / max(n_pred, n_gt)
**元素集合相似度 = Jaccard相似度 = |交集| / |并集|
***环数相似度 = 1 - |ring_pred - ring_gt| / max(ring_pred, ring_gt, 1)

**惊人的发现**：
- Rank3 的真实结构相似度最高（0.89），但检索分数最低（0.2357）
- 检索分数与结构相似度呈**负相关**！
- 说明检索头学到的是 AFM 图像相似度，而非分子结构相似度

## 四、根本原因总结

### 4.1 直接原因

1. **原子数量预测严重偏差**
   - 预测 36 个 vs 真实 21 个（+71%）
   - 导致检索偏向大分子

2. **原子类型预测错误**
   - 完全缺失 3 个氧原子（占14%）
   - type_loss 权重过低（0.1）

3. **环结构信息丢失**
   - 原子对应关系错乱，环检测失败
   - 检索特征不包含拓扑信息

### 4.2 系统性问题

1. **检索目标与生成目标不一致**
   - 检索基于 AFM 特征
   - 生成基于扩散模型
   - 两者没有端到端训练

2. **特征空间设计不合理**
   - 128 维无法表示 21 万个复杂分子
   - 缺少原子数、元素类型、环结构等结构化信息

3. **验证集训练缺失**
   - 验证集分子 CID 不在训练集中
   - retrieval_loss = 0.0，无监督信号
   - 模型无法学习泛化

## 五、改进方案

### 5.1 短期方案（1-2周，预期相似度 0.4-0.5）

#### 方案A：改进原子数量预测

```python
# 1. 增强 AtomCountHead
class ImprovedAtomCountHead(nn.Module):
    def __init__(self, embed_dim=512, max_count=85):
        super().__init__()
        # 更深的网络
        self.encoder = nn.Sequential(
            nn.Linear(embed_dim, 512),
            nn.LayerNorm(512),
            nn.GELU(),
            nn.Dropout(0.1),
            nn.Linear(512, 256),
            nn.LayerNorm(256),
            nn.GELU(),
        )
        # Focal Loss 处理类别不平衡
        self.cls_head = nn.Linear(256, max_count)

# 2. 增加 count_loss 权重
losses["loss"] = (
    coord_loss
    + 0.1 * type_loss
    + 1.0 * count_loss  # 从 0.5 提升到 1.0
    + 0.05 * retrieval_loss
)
```

**预期效果**：
- 原子数量准确率从 44.65% 提升到 60%+
- MAE 从 0.79 降低到 0.5
- 检索到的分子原子数更接近真实值

#### 方案B：增加检索特征维度

```python
class MoleculeRetrievalHead(nn.Module):
    def __init__(self, embed_dim=512, proj_dim=512):  # 从128提升到512
        super().__init__()
        self.projector = nn.Sequential(
            nn.Linear(embed_dim, 512),
            nn.GELU(),
            nn.Linear(512, proj_dim),
        )
```

**预期效果**：
- 检索空间从 128 维扩展到 512 维
- 每个维度区分的分子数从 1650 降低到 413
- 相似度分数提升 10-20%

#### 方案C：多阶段检索

```python
def retrieve_with_filters(afm_features, n_atoms_pred, top_k=5):
    """带结构过滤的检索"""
    # 1. 粗筛：原子数过滤
    candidates = filter_by_atom_count(
        all_molecules,
        target=n_atoms_pred,
        tolerance=3  # ±3 个原子
    )

    # 2. 精筛：特征相似度
    scores = compute_similarity(afm_features, candidates)

    # 3. 返回 Top-K
    return sorted(candidates, key=lambda x: scores[x])[:top_k]
```

**预期效果**：
- 过滤掉明显不相关的分子（如35个原子的不会匹配21个的）
- 相似度分数提升到 0.4-0.5

### 5.2 中期方案（1-2月，预期相似度 0.5-0.7）

#### 方案D：基于生成结果的检索

```python
def retrieve_from_generated_structure(coords, atom_types, top_k=5):
    """从生成的结构计算分子指纹并检索"""
    # 1. 计算分子指纹（如 ECFP, Morgan fingerprint）
    fingerprint = compute_molecular_fingerprint(coords, atom_types)

    # 2. 在数据库中检索相似分子
    from rdkit import Chem, DataStructs
    similar_cids = search_by_fingerprint(fingerprint, top_k=5)

    return similar_cids
```

**优点**：
- 使用真实的分子结构信息
- 包含拓扑、键连接、环结构
- 不依赖 AFM 特征

**缺点**：
- 需要 RDKit 集成
- 依赖生成质量

#### 方案E：层次化检索架构

```
第一层（粗粒度）：结构特征过滤
  - 原子数：21 ± 3
  - 元素类型：必须包含 {O, N, C, H}
  - 质量范围：计算分子量
  ↓
候选池：211,505 → ~5,000

第二层（中粒度）：拓扑特征匹配
  - 环数量：2 ± 1
  - 官能团：检测羧基、氨基等
  - 芳香性：芳香环 vs 脂肪环
  ↓
候选池：5,000 → ~500

第三层（细粒度）：3D结构排序
  - 分子指纹相似度
  - 3D形状匹配（如 Shape-based alignment）
  - AFM 特征余弦相似度
  ↓
Top-5 结果
```

**预期效果**：
- 相似度分数 > 0.6
- 结构相似度与检索分数一致

### 5.3 长期方案（2-3月，预期相似度 0.7-0.9）

#### 方案F：端到端联合训练

```python
class JointReconstructionRetrieval(nn.Module):
    def __init__(self):
        super().__init__()
        self.encoder = VideoViTEncoder()
        self.ddpm = ConditionalDDPM()
        self.retrieval_head = MoleculeRetrievalHead()

    def forward(self, batch):
        # 1. 编码
        c = self.encoder(batch["afm_stack"])

        # 2. 生成
        coords, types = self.ddpm.sample(c, batch["n_atoms"])

        # 3. 从生成结果提取特征用于检索
        structure_features = extract_structure_features(coords, types)

        # 4. 联合检索（AFM特征 + 结构特征）
        retrieval_features = torch.cat([c, structure_features], dim=-1)
        retrieval_scores = self.retrieval_head(retrieval_features)

        return coords, types, retrieval_scores
```

**优点**：
- 检索和生成共享信息
- 结构特征直接用于检索
- 端到端优化

## 六、实验建议

### 6.1 诊断实验

**实验1：理想情况下的检索性能**
- 输入：真实的原子数、原子类型（而非预测值）
- 计算分子指纹
- 在数据库中检索
- **目的**：测试检索任务的上限

**实验2：特征维度的影响**
- 测试 128 / 256 / 512 / 1024 维检索空间
- 评估相似度分数变化
- **目的**：确定最优维度

**实验3：多模态特征融合**
- AFM 特征 + 预测原子数 + 预测元素分布
- 拼接后进行检索
- **目的**：验证结构化信息的价值

### 6.2 评估指标

除了余弦相似度，还应评估：

1. **结构相似度指标**
   - 原子数 MAE
   - 元素 Jaccard 相似度
   - 环数 MAE

2. **Top-K 准确率**
   - Top-1 命中率（检索到的第一个分子是否结构相近）
   - Top-5 召回率（Top-5 中至少一个结构相近）

3. **检索-生成一致性**
   - 检索到的分子与生成的分子的 RMSD
   - 检索到的分子与真实分子的 RMSD 对比

## 七、结论

### 核心发现

1. **相似度 0.24 的真正含义**：
   - 不代表分子结构相似度
   - 代表 AFM 图像特征空间的相似度
   - 受原子数量和类型预测错误影响

2. **最大的问题**：
   - 原子数量预测错误（+71%）
   - 检索维度不足（128 维 vs 21 万分子）
   - 检索与生成割裂（两个独立任务）

3. **最有潜力的改进方向**：
   - 短期：提升原子数量预测 + 增加检索维度
   - 中期：基于生成结果的检索
   - 长期：端到端联合训练

1. ✅ 改进 AtomCountHead（方案A）
2. ✅ 增加 count_loss 权重到 1.0
3. ✅ 扩展检索维度到 256 或 512（方案B）
